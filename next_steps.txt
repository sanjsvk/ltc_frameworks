Notes for step 4 -
Maximise recovery rate per scenario per model. Simple, interpretable, directly maps to the paper's metric. Optimise each model independently per scenario.

F1 — Static Adstock (4 models)
Strategy: Grid search + cross-validation
Reason: F1 models are fast to fit, few parameters, grid search is exhaustive and reproducible

For geo_adstock and dual_adstock:
  - Expand decay grids to full range [0.10, 0.95] in 0.05 steps
  - Per channel, per scenario
  - Select decay that maximises recovery rate
  - Log selected values — these become the scenario-specific optimal parameters

For weibull_adstock:
  - Scipy optimize (L-BFGS-B) on shape and scale per channel per scenario
  - Objective: minimise LTC MAPE with channel ranking constraint
  - Multiple random restarts (5-10) to avoid local minima
  - Bounded: shape [0.5, 3.0], scale [1.0, 20.0]

For almon_pdl:
  - Grid search on degree [2, 3, 4] and max_lag [13, 26, 39, 52] per channel
  - Separate STC and LTC polynomial degrees
  - Select combination maximising recovery

F2 — Dynamic Distributed Lag (3 models)
Strategy: AIC-guided search + regularisation tuning
Reason: F2 models have more parameters, AIC selection is appropriate

For koyck:
  - Grid search on lambda per channel (same as F1 approach)
  - Additionally tune AR order [1, 2, 3, 4]
  - Select lambda + AR order combination minimising AIC subject to recovery constraint

For ardl:
  - Tune ltc_degree [2, 3, 4] and ltc_max_lag per channel
  - Tune AR order [1, 2, 3]
  - Key: add L2 regularisation on lag coefficients to prevent sign-flip
  - Regularisation strength tuned via cross-validation on held-out weeks

For finite_dl:
  - Scipy optimize on shape and scale per channel per scenario
  - Same approach as weibull_adstock
  - Additionally tune stc_cutoff [4, 6, 8]

F3 — State-Space (3 models)
Strategy: Empirical Bayes + MLE profile likelihood
Reason: F3 models are expensive — need targeted search not brute force

For kalman_dlm:
  - Profile likelihood over delta [0.80, 0.85, 0.88, 0.90, 0.92, 0.95]
  - For each delta value, MLE estimates remaining parameters
  - Select delta maximising log-likelihood subject to recovery constraint
  - Additionally tune level_var [0.001, 0.005, 0.01, 0.05] — this is under-explored

For bsts:
  - Same profile likelihood approach on delta
  - Additionally tune seasonal_var [0.0001, 0.001, 0.01]
  - Tune slope_var [0.00001, 0.0001, 0.001]
  - These variance parameters control baseline flexibility — key for S3/S4

For mcmc_stock:
  - Empirical Bayes: use MAP estimates from S1 as starting priors
  - Per scenario, update prior means based on scenario characteristics
    S2: widen build_rate prior (spend pause makes it less identifiable)
    S3: tighten seasonal component (rich seasonal signal available)
    S4: update obs_sigma to post-break spend level
  - Tune target_accept [0.90, 0.95, 0.97] and tune steps [500, 1000, 2000]

Logging structure for optimisation results
Keep optimised parameters completely separate from the original frozen parameters. This is critical for the paper — the frozen parameter results are your main results, the optimised results are the sensitivity analysis.

Recommended structure - 
optimised_parameter_log.csv
  exp_id, scenario, model, parameter, optimised_value, original_value,
  recovery_improvement_pp, mape_improvement_pp, channel_ranking_correct

optimised_results_summary.csv
  scenario, model, original_recovery, optimised_recovery, improvement_pp,
  original_mape, optimised_mape, key_parameter_changed

The gap between frozen-parameter recovery and optimised-parameter recovery is the paper's calibration sensitivity finding. A large gap means the model is highly sensitive to parameter choice — risky for practitioners. A small gap means the model is robust to calibration — reliable for practitioners

Expected pattern:

F3 methods: small gap (5–15pp improvement) — structurally robust, calibration helps modestly
F2 methods: medium gap (10–25pp improvement) — moderate calibration sensitivity
F1 methods: large gap or no improvement (0–30pp) — either highly calibration-sensitive or structurally incapable regardless of calibration

This pattern, if confirmed, adds a fifth dimension to the paper's framework comparison: not just which framework performs best, but which framework is most robust to sub-optimal calibration — which is always the practitioner's real-world condition.
